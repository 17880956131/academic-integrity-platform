# 论文智能审核系统 — 项目白皮书

## Academic Integrity Platform v3.0 · 为什么需要 + 设计思路

---

## 一、为什么需要这个系统？

### 1.1 学术诚信危机：一个正在瓦解的地基

2024-2026 年间，全球学术界面临三重前所未有的冲击：

**第一重冲击：AI 生成论文的指数级泛滥。**

自 GPT-4、Claude 等大语言模型向公众开放以来，学术论文的"作者"不再必然是人类研究者。据《Nature》2024 年调查，在提交给顶级期刊的稿件中，约有 6-17% 的段落被检测出 AI 生成特征。更令人担忧的是，现有的 AI 检测工具（如 GPTZero、Turnitin）对经过人工改写润色的 AI 文本，误判率高达 15-25%。

**第二重冲击：抄袭手段的"军备竞赛"升级。**

传统查重系统依赖字符级匹配和 TF-IDF 余弦相似度，面对以下新型抄袭手段几乎失效：

- **同义改写式抄袭**：用 LLM 重写整个段落，保留原始语义但消除字面匹配
- **翻译回译式抄袭**：中文论文 → 英文 → 法文 → 中文，字符序列完全改变
- **拼图式抄袭**：从 5-8 篇源文献各取一段，语义缝合度极高
- **图表数据抄袭**：在不改变图表外观的情况下，直接挪用实验数据

**第三重冲击：创新优先权的"无主时代"困境。**

当前的学术评价体系存在一个致命的时序盲区：当一个创新性想法同时被两个团队独立提出时，谁拥有"优先权"？传统方案依赖期刊的收到日期戳，但这一机制面临：

- 投稿到出版长达 6-18 个月的时间窗口
- 预印本服务器的非正式性与不可追溯性
- 缺乏不可篡改的时间戳证据链

**这三重冲击共同指向一个结论：传统的"文字查重"式学术诚信检测，已经无法适应 AI 时代的学术生态。我们需要一个多维度的、从"查相同"到"证不同"的范式转型。**

---

### 1.2 学术诚信检测的范式缺陷

| 传统范式 | AI 时代失效原因 | 本系统方案 |
|---------|----------------|----------|
| **文字查重**（字符匹配） | 同义改写/翻译回译/LLM重写可轻易绕过 | **三重检测体系**：文字层 + 语义层 + 逻辑层 |
| **单一 AI 检测**（perplexity 阈值） | 经过人工润色的 AI 文本难以分辨 | **20+ 模型指纹库**：匹配 burstiness/熵分布等多维特征 |
| **期刊日期戳**（投稿时间） | 非即时、可伪造、无法律效力 | **区块链毫秒级时间戳** + SHA-256 哈希存证 |
| **同行评审**（核心创新判断） | 主观性强、周期长、无量化指标 | **ECU 创新势能模型**：双函数数学量化 |
| **术语首用认定**（文献回顾） | 碎片化、跨语言追踪困难 | **知识图谱本体锚点** + 语义溯源 |

---

### 1.3 系统设计目标

本系统旨在构建一个**从"检测抄袭"到"保护创新"的完整闭环**：

```
传统系统: 提交论文 → 查重 → 返回相似度 → 结束
本系统:   提交论文 → 七维检测 → 区块链存证 → 创新势能评估 → 争议仲裁 → 创新确权
```

**核心设计原则**：

1. **从"否定式"到"肯定式"**：不只告诉你"抄袭了哪里"，更告诉你"创新在哪里"
2. **从"事后检测"到"实时确权"**：在论文完成的第一时间就进行区块链存证，而非等到投稿
3. **从"单一工具"到"生态平台"**：覆盖提交→检测→存证→仲裁→创新的完整生命周期
4. **从"黑箱判断"到"可解释结论"**：每个检测结果都有证据链，每条仲裁都有权重计算

## 二、系统设计思路

### 2.1 架构哲学：七维检测体系

系统设计了七个相互独立又层层递进的检测维度，形成完整的多维分析矩阵：

```
        文字层（基础）
         ↓ 验证
        语义层（AI检测 + 术语溯源）
         ↓ 深化
        逻辑层（创意验证 + ECU 创新势能）
         ↓ 固化
        证据层（区块链存证 + 脑认知指纹）
         ↓ 保障
        仲裁层（争议仲裁机制）
```

**每一层解决一个不可回避的学术诚信问题**：

| 层级 | 检测模块 | 解决的核心问题 | 核心技术 |
|:---:|---------|--------------|---------|
| L1 | 文字查重 | "这段文字和别人一样吗？" | TF-IDF + 知网/万方API |
| L2 | AI生成检测 | "这是人写的还是AI写的？" | Perplexity + Burstiness + 20模型指纹 |
| L2 | 术语溯源 | "这个术语是谁最先用的？" | Neo4j 知识图谱 + 本体锚点 |
| L3 | 创意验证 | "这个想法是原创的吗？" | Claude LLM 解构 + 虚拟模拟 |
| L3 | ECU 创新势能 | "这条创新有多少势能？" | 双函数数学建模 |
| L4 | 区块链存证 | "怎么证明我是第一个？" | SHA-256 + 毫秒时间戳 |
| L4 | 脑认知指纹 | "是我本人大脑创作的吗？" | EEG 波形特征（接口预留） |
| L5 | 争议仲裁 | "两个人都说是自己原创怎么办？" | 证据权重体系 + 仲裁规则 |

### 2.2 前端设计：零依赖 SPA 的工程选择

#### 为什么选择纯原生 JavaScript？

| 考量维度 | React/Vue 方案 | 纯原生方案（本系统） |
|---------|:---:|:---:|
| 学习成本 | 需理解 JSX/V-DOM/响应式 | 标准 ES6 即可 |
| 构建工具 | npm + webpack/vite | 零构建，直接运行 |
| 服务器需求 | 需要 Node.js | 任意静态文件服务 |
| 文件大小 | 200KB+（含框架） | <80KB 总代码 |
| 浏览器兼容 | 需 polyfill | ES6 原生支持 |
| 部署复杂度 | 构建 → 优化 → CDN | 复制粘贴即部署 |
| 学术用户获取 | 需要安装环境 | 打开即用 |

**这是一个面向学术界而非前端开发者的系统。** 让一个化学教授下载 zip 包、安装 Python 3 就能在本地跑起来，比让他配 node/npm/webpack 更合理。

#### 设计令牌（Design Tokens）体系

```css
/* 12 个语义色 — 每个都有明确的语义映射 */
--blue:   #185FA5  /* 数据驱动：链接、主按钮、选中态       */
--green:  #3B6D11  /* 正向反馈：通过、安全、认证通过       */
--red:    #A32D2D  /* 风险警示：抄袭、AI生成、高风险       */
--amber:  #854F0B  /* 注意提示：中等风险、待处理、模糊区域  */
--purple: #534AB7  /* 创新维度：创意验证、ECU模型、前瞻性  */
--teal:   #0F6E56  /* 技术维度：区块链、存证、技术指标     */

/* 5 级灰阶 — 信息层级而非装饰 */
--bg:     #F8F9FA  /* 页面背景 */
--bg-card: #FFFFFF /* 卡片背景 */
--border: #DEE2E6  /* 分割线 */

--hint:   #6C757D  /* 辅助文字 */
```

**设计决策**：不使用渐变色、不使用发光阴影、不使用圆角卡片风。原因：

- 学术工具的视觉语言应该是**严谨**而非花哨
- 低视觉干扰让用户聚焦于数据判断
- 颜色仅用于传达信息状态（红=风险、绿=安全、紫=创新）

#### Hash 路由设计

```
#/dashboard        → 检测总览（默认首页）
#/submit           → 论文提交（4步提交流程）
#/report?id=xxx    → 检测报告（6个Tab）
```

**设计原则**：
- 每个检测维度都有独立入口（不仅通过报告页Tab访问）
- URL 可分享、可书签、可回溯
- 每个页面独立渲染，不依赖其他页面的状态
- 导航按"核心功能 / 进阶分析 / 系统"三组分类

### 2.3 数据设计：Mock 驱动的渐进式开发

系统中的所有数据均通过 `mockData.js` 模拟，但数据结构严格遵循真实 API 规范：

```javascript
// 模拟数据 = 真实数据结构的预览
mockData.recentPapers = [
  {
    id: "PAPER-2026-0001",
    title: "基于深度迁移学习的跨领域知识图谱构建方法研究",

    detection: {
      aiDetection: { probability: 0.34, matchedModels: [...] },
      plagiarism: { overallRate: 0.12, sources: [...] },
      creative: { noveltyScore: 0.78, lifecycle: [...] },
      terminology: { registeredTerms: 3, conflicts: 0 },
      blockchain: { hash: "...", timestamp: "..." },

```

**这种设计使得切换到真实后端时，前端代码零修改**——只需将 `mockData.xxx` 替换为 `fetch('/api/v1/xxx')`。

### 2.4 后端引擎：三层降级架构

```
API 调用（优先）
  ├─ 成功 → 返回高精度结果 + 缓存 7 天
  └─ 失败/超时
      └─ 本地引擎（降级）
          ├─ 有本地方案 → 返回标注「精度受限」的结果
          └─ 无本地方案 → 返回 partial_result + 明确标注
```

**降级策略的核心思想**：宁可返回不完整的结果，也不返回错误的结果；宁可标注"此环节精度受限"，也不假装检测了。

| 引擎 | API | 降级 | 降级精度 | 不可降级 |
|------|:---:|:---:|:---:|:---:|
| 查重 | 知网 | TF-IDF | ~70% | — |
| AI检测 | GPTZero | RoBERTa | ~85% | — |
| 创意验证 | Claude | — | — | ⚠️ 标记 |
| 术语溯源 | Embedding | Neo4j | ~85% | — |
| 区块链 | — | SHA-256 | 100% | — |
| ECU | — | 数学公式 | 100% | — |

### 2.5 ECU 创新势能：从"定性评审"到"定量建模"

ECU（Effective Creative Unit）是本系统最核心的理论创新。它源于一个简单的观察：

> 当前的学术评价中，"创新性"是一个模糊的、不可量化的概念，主要依赖同行评审的主观判断。但如果我们能将其分解为几个可测量的维度，就能构建一个数学上自洽的评估模型。

**ECU 双函数模型**：

```
ECU(x) = α · Novelty(x) + β · Impact(x) + γ · Feasibility(x)

其中：
  Novelty(x)     = 1 − max(sim(x, corpus_i))     // 与现有语料的最大不相似度
  Impact(x)      = log(1 + citation_potential)     // 对数缩放的引用潜力
  Feasibility(x)  = σ(methodology_rigor)           // 方法论严谨度的 Sigmoid 映射
  α = 0.5, β = 0.3, γ = 0.2                       // 权重（可调整）
```

**为什么需要这个模型？**

1. **跨学科可比性**：一个数学证明的新颖性和一个实验方法的新颖性，通过 ECU 可以在同一尺度上比较
2. **争议仲裁的客观依据**：当两个相似发现产生争议时，ECU 密度差异可作为判断独立发现的依据
3. **创新趋势分析**：通过 ECU 密度的时间序列，可以识别一个学科领域的创新高峰和瓶颈期

### 2.6 争议仲裁：创新优先权的数学证明

这是系统中最具法律工程思维的设计。当两个研究者声称对同一发现拥有优先权时：

**不是靠"谁先投稿"，而是靠"谁先存证 + 证据链完整性"来判断。**

```
仲裁公式:
  优先权得分 = 0.40 × 区块链存证时间 + 0.30 × 脑认知指纹匹配度
             + 0.20 × 文档修改历史 + 0.10 × 术语首次注册时间

证据权重来源:
  40% — 区块链存证（不可篡改的毫秒级时间戳）
  30% — 脑认知指纹（创作时的 EEG 特征与论文内容的关联性）
  20% — 文档修改历史（Git/版本管理中的渐进式修改记录）
  10% — 术语优先注册（在知识图谱中首次锚定的时间）
```

这套规则源于 IEEE 学术区块链工作组的讨论草案，在本系统中被实现为可执行的计算逻辑。

## 三、从设计到实现的工程决策

### 3.1 为什么先做前端 SP

A，而不是先写后端？

1. **后端依赖外部 API**：知网、GPTZero、Claude 都需要Key和服务协议，不是纯开发问题
2. **前端即产品**：Mock 数据驱动的 SPA 就是可演示、可体验的完整产品
3. **接口先行**：前端的数据结构定义了后端 API 的 Contract，后端实现只是填空
4. **降低贡献门槛**：前端纯 HTML/CSS/JS，非程序员（如学术编辑）也能提出修改意见

### 3.2 后端引擎的"骨架先行"策略

`backend/engines/detection_engine.py` 并非一个可运行的完整后端，而是：

- 定义了每个引擎的**接口契约**（输入/输出数据结构）
- 实现了**纯本地计算**的部分（SHA-256、ECU模型）
- 为外部 API 调用留下了**清晰的接入点**（参数、格式、错误处理）
- 是后续贡献者的**开发地图**——每个 `TODO` 都是一个可独立完成的任务

### 3.3 开源的渐进策略

```
Phase 1（已完成）: 前端 SPA + 后端骨架 + 设计文档
         ↓
Phase 2（社区贡献）: 接入 1-2 个真实 API（GPTZero + Claude）
         ↓
Phase 3（生产可用）: 完整 7 引擎 + Redis 缓存 + 异步队列
         ↓
Phase 4（生态扩展）: 术语知识图谱社区共建 + 区块链联盟节点
```

## 四、总结

### 4.1 这个系统在解决什么？

**不是在做一个"更强的查重软件"。** 而是在构建一个"创新"可以像土地一样被测量、登记、确权的数字基础设施。

### 4.2 核心创新点

| 创新维度 | 传统方案 | 本系统方案 |
|---------|---------|----------|
| 检测范围 | 文字查重（1维） | 七维检测体系 |
| AI 文本 | 单一阈值判断 | 20+ 模型指纹库 + 三重检测 |
| 创新评估 | 同行评审（主观） | ECU 数学建模（量化） |
| 优先权认证 | 期刊日期戳（可伪造） | 区块链毫秒时间戳（不可篡改） |
| 争议处理 | 编辑裁决（人为） | 证据权重计算（算法辅助） |
| 部署方式 | 商业 SaaS | 开源 + 自托管 |

### 4.3 适合谁？

- **高校学术诚信办公室**：替代昂贵的商业查重系统
- **期刊编辑部**：论文投稿前的多维预审
- **独立研究者**：在论文公开发布前完成创新确权
- **学术政策研究者**：研究 AI 时代学术评价机制的基础设施
- **开发者**：基于骨架引擎接入自己的检测模型
